Context

IAM modernization and the SSH Service

What’s happening

  • The adoption of portal.cscs.ch required a modernization of the IAM services and tooling around it
  • The SSH Service is a core service: it’s how users access the centre
  • Over the last months, IAM completed a full refactoring: backend API, web frontend, and CLI
  • The new version has been in production for a few weeks, currently for internal use

Today’s session

  • We’ll show you the new tooling and the main differences
  • We’ll do live demos of the Webapp and the CLI
  • We want to hear your feedback and evaluate possible improvements

About the legacy service

  • The old SSH Service (sshservice.cscs.ch) and sshservice-cli are still active and working
  • No retirement date has been set yet
  • We will evaluate the timeline in the coming weeks, also based on your feedback
  • Today is about showing you what’s new. No urgency to migrate right now
Your feedback matters
This is also an opportunity for us to collect feedback and prioritize improvements based on your real-world needs.
The core concept is unchanged: users access CSCS with CSCS-signed SSH keys. What’s new is the tooling. The recommended flow is now: generate your key locally, then sign it. Your private key never leaves your machine.

What should users use now?

New entry points for SSH-related workflows

User Account Webapp

  • Sign an existing local key (recommended, your private key never leaves your machine)
  • Or generate a new key pair server-side (deprecated, retirement date TBD)
  • List, inspect, and revoke your SSH keys
  • The recommended web entry point for SSH workflows
user-account.cscs.ch

New CLI

  • cscs-key sign sign your local public key (recommended)
  • cscs-key list list your active SSH keys
  • cscs-key revoke revoke a key by serial number
  • A convenience for shell-based workflows. For custom integrations, use the API directly
github.com/eth-cscs/cscs-key
Recommended flow: generate your key locally, then sign it via Webapp or CLI. Your private key never leaves your machine.  ·  Docs: docs.cscs.ch/access/ssh/
Note: cscs-key binary signing. The CLI distributes pre-built binaries for Linux, macOS, and Windows. Some users may see security warnings because the binaries are not yet signed. Signed binaries will be available soon. Please let us know if you need support.

Which account should I use?

A quick practical view for end users

Standard User
Personal access to CSCS systems

Use it when

  • You need human access to Alps for daily project work

Main traits

  • Human account
  • Personal access
  • Can span multiple projects
Service Account
Automation & CI/CD

Use it when

  • Pipelines, CI/CD, or any unattended workflow

Main traits

  • Non-human account
  • Project-scoped
  • Ephemeral keys only (1 min)
Temporary Account
Short-lived event access

Use it when

  • Courses, trainings, workshops, hackathons

Main traits

  • Short-lived, event-specific
  • Simpler onboarding, expires automatically
Service accounts require a request to IAM via support ticket. Specify: project name, number of accounts needed, and use case. Once approved, the PI / Deputy PI can create them on portal.cscs.ch.
Rule of thumb:  logging in as yourself → Standard User  ·  pipelines or unattended jobs → Service Account  ·  course or training → Temporary Account
